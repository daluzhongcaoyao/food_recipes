问题：

我打算写一个查询和管理食谱的前端网站，我需要你帮我做下架构设计。
最终是需要用docker compose部署在1c1g的Ubuntu vps

问题：
1、我应该写静态网站还是前后端网站
2、我应该直接写html还是用vue3
3、是否需要数据库
4、搜索怎么实现

需求：
1、分为查询和管理，这应该是通过2个tab来切换的
2、食谱本身，是1个图片，在管理页面应该是能直接上传的。上传后保存在服务器的index.html所在目录下（最好是子目录，如food文件夹）
3、新增和编辑食谱的时候，支持添加删除和修改tag，tag我一般是填食材，作者
4、在查询页面，支持输入tag进行筛选，多个tag取交集
5、查询页面，左边是食谱列表（顶部支持输入tag进行筛选，输入框需要支持联想当前已有tag，按回车后确认tag，并且可以继续输入其他tag，tag重复的就不添加。添加的tag显示在搜索框下面。tag下面是所有符合要求的食谱列表）。右边是食谱（食谱是个16:9的图片，底部显示tag，点击tag可以将其添加到左侧的tag筛选）。在食谱列表点击任意食谱，右侧显示对应的食谱图片和tag。
6、左右两侧各留1/12的空白，左边栏宽度是1/6

=====================
回答：

这是一个非常具体且有趣的个人项目。考虑到你的目标是部署在 1核1G内存（1C1G） 的低配VPS上，资源利用率是架构设计的核心考量。

以下是针对你问题的详细分析和架构建议：

核心决策分析
1. 静态网站 vs 前后端分离
结论：必须是前后端分离（或轻量级全栈）。

原因： 你的需求里有“管理”功能（上传图片、新增/修改数据）。纯静态网站（只有HTML/JS/CSS）无法处理文件上传和数据持久化，必须有一个后端服务来接收图片并保存数据。
2. 直接写 HTML vs Vue3
结论：强烈推荐 Vue3。

原因：
交互复杂度： 你的“查询页面”需求很复杂（Tag联想、多Tag交集筛选、左右分栏联动、点击Tag反向筛选）。用原生JS/jQuery写这些逻辑会非常痛苦且难以维护。Vue3 的响应式数据绑定能轻松处理这些状态变化。
组件化： 你的“食谱卡片”、“Tag输入框”都是可复用的组件。
3. 是否需要数据库
结论：不需要传统数据库（如MySQL），使用 JSON 文件即可。

原因：
资源限制： MySQL 或 PostgreSQL 即使空跑也会占用 200MB+ 内存，对于 1G 内存的机器太奢侈了。
数据量： 个人食谱通常在几百到几千条，完全可以存放在一个 data.json 文件中。
性能： Node.js 读取几MB的 JSON 文件并在内存中进行筛选，速度极快（微秒级），远快于数据库查询。
4. 搜索怎么实现
结论：前端内存搜索 或 后端内存搜索。

方案： 既然数据量不大（假设 < 5000 条），最简单的做法是前端一次性加载所有食谱的元数据（JSON），然后在浏览器端直接用 JS 的 filter 函数进行 Tag 交集筛选。这样响应速度最快，体验最好。
架构设计方案：轻量级全栈方案
为了适应 1C1G 的环境，我推荐使用 Vue3 (前端) + Node.js/Koa (轻量后端) 的组合。

技术栈选型
前端框架： Vue 3 + Vite + TailwindCSS (样式编写快，体积小) + Element Plus (或 Naive UI，组件库方便做Tag输入和上传)。
后端服务： Node.js (Koa 或 Express)。
数据存储： recipes.json (本地文件)。
图片存储： 本地文件系统 (/public/uploads 目录)。
部署： Docker Compose (Nginx + Node App)。
